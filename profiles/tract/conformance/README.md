# TRACT Conformance

## What this directory is for

§15 defines conformance as passing a suite of cases, not resembling the reference
implementation (GOVERNANCE.md, "Implementation neutrality"). This directory is meant to hold
that suite: a case catalogue and, once there is something to freeze, byte-exact known-answer
vectors — so that two implementations that have never seen each other's code can check they
agree with the specification, not merely with each other.

## The discipline, stated before there is anything to apply it to

A vector corpus that an implementation passes proves the corpus and the implementation agree
with **each other**. It does not prove either one agrees with the **specification**. That
distinction sounds pedantic until it fails, and it has already failed once, on the sibling
DMTAP spec this project is built on: a review found a case where the spec text, the frozen
vector, and three tests were all self-consistently wrong **together**. Nothing in that
arrangement could have caught the error, because the vector had been produced from the same
reference implementation the tests ran against — so implementation and corpus checked each
other in a closed loop, and the loop had no path back to the prose.

The fix DMTAP settled on, and the rule TRACT inherits: **a vector is derived by someone reading
the specification text, not exported from a running implementation.** A generator script that
computes bytes from the same formulas the spec states, run by a person checking those formulas
against the section they cite, is sound. A dump of a reference implementation's internal state,
however convenient, is not — it can only ever prove that implementation self-consistent, which
is a fact about the implementation, not about the document. Where a reference crate does produce
vectors, an independent, from-scratch re-derivation that deliberately does not import the
generator is what actually closes the loop back to the spec (DMTAP's
`conformance/vectors/verify_pub_vectors.py` is the pattern to follow when TRACT reaches that
stage).

This also follows directly from this repository's own governance: "no implementation is part of
the standard" (`GOVERNANCE.md`), and "where an implementation and this document disagree, this
document wins." A vector exported from Soko would quietly make Soko normative by the back door,
regardless of what the governance document says on paper.

## Current state: zero vectors

There are no conformance vectors in this directory, and the absence is deliberate rather than an
oversight. §16 (wire format) — the deterministic CBOR object definitions for `ProductRecord`,
`Offer`, `Order`, `EscrowScope`, `RateCard` and the rest — is not yet normative (see the
drafting-status banner on `16-wire-format.md`). Nothing in §16 has committed byte shapes yet: key
numbering, canonicalisation rules and field sets are all still open questions in §2–§14's stub
prose. There is, in the literal sense, nothing to freeze.

Freezing vectors ahead of §16 would do one of two things, both bad:

- invent byte shapes that no section actually specifies, making this directory a de facto
  specification written in JSON instead of prose, reviewed with none of the scrutiny the
  numbered sections get; or
- get silently invalidated the moment §16 lands with a different shape, and nobody would notice,
  because a vector nothing runs cannot fail a build.

`conformance/SUITE.md` in this directory is therefore a **case catalogue, not a corpus**: it
enumerates the invariants a future suite will need to check, organised by section, every one of
them marked PLANNED. It commits to no bytes, no counts of what is runnable, and no schedule.

## What changes when §16 goes normative

The discipline above is not a placeholder for later — it is the rule that governs the first
vector as much as the hundredth:

1. A vector is written from the §16 text of the object it covers, by someone who read that text,
   not generated by pointing a script at Soko and recording what came out.
2. Soko's own test suite is expected to re-derive the same vectors independently and compare —
   the cross-check that catches drift, not the source of truth.
3. A divergence between Soko and a committed vector is investigated as "which one is wrong,"
   with the specification text as the tiebreaker. It is never resolved by editing the vector to
   match Soko, which is the exact failure mode this section exists to name.

Until then, this directory holds discipline and scope, not test data. `make coverage` in the
repository root reports the honest current numbers rather than a projected one.
